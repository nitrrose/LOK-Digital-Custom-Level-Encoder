LOK DIGITAL CUSTOM LEVEL DATA:
¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯¯

My research into the LOK Digital custom level encoding system, as part of my project to create a random level generator.

#----------------------------------------------# Dictionary

trixel = a 3x3 pixel in the data
empty = a tile with no attached data
GCV (grey colour value) n = the RGB value corresponding to R:n B:n G:n
iced blocks = featured in World A, allows for (n) additional uses of a tile before it can be blacked out, where (n) is the number in the corner of the block
arrows = featured in World C, move the tile in the pointing direction once spelt, not blacked out. Also works from question marks

#----------------------------------------------# Reading the data

• The data strip is read from left-to-right, bottom-to-top.
• It is 64 trixels long and 1 or more trixels tall, and is surrounded by a pure white border of width 1 pixel.

• If the data is too long, it expands upward until all the data can be fit into the number of trixels.
• Any leftover space is filled in with pure black trixels.

• Tile data is grouped into 10 sets of 11 trixels, a prefix and a suffix. The prefix and suffix are for non-tile data, such as modifiers.

• The prefix appears to always be this sequence of trixel GCVs: <0 0 0 {# of trixels in strip} {ascii digits of row size} 10 {ascii digits of col size}>.					

• The suffix appears to always be this sequence of trixel GCVs: {10 36} followed by the arrows and cloud modifier toggles.

• Each set begins with a trixel with a GCV of 10, before the tile data continues.
• The first trixel is the first tile in the row, and each set corresponds to a different row, such that row 1 = set 1, row 2 = set 2, etc. from the top of the screen (i.e. top row = row 1). 
• Some tiles span 2 trixels, such as iced blocks.

• The trixels only check for the red channel of the colour. Green and Blue channels can hold any value, but the red holds the tile's data (the decimal number).

#----------------------------------------------# Decoding and encoding data

• Letters are the GCV for their ASCII code, e.g.
	A = 65 = GCV 65
	B = 66 = GCV 66
	C = 67 = GCV 67
	       …

• Exceptions such as arrows, question marks, blanks and empties have these characters:
	↓   = 118
	↑   = 94
	→   = 62
	←   = 60
	?   = 63
	! = 33
	blank = 35
	empty = 45
	+ = 43
	, = 44
	. = 46
	: = 58
	blacked out tile = 61
	\n = 10
	- !!note that this is a known parasite value!!
	- a parasite value is a tile which when introduced into leveldata will cause the level to persist when you exit the editor. when you return, it will still be there and causes many bugs

• Iced blocks are encoded in this form:
	[Letter on tile : Number on ice]

• The letter encoding follows the same rules as before, but the number on ice follows this pattern:
	1 = 49
	2 = 50
	3 = 51
	   …
	10 (0) = 48
	
	note that these numbers can also be input into tiles by themselves

• Modifiers (e.g. clouds/arrows) are denoted in the suffix. Arrows are the 2nd trixel and Clouds are the 3rd trixel.
• If a modifier is enabled, the pixel colour is RGB 49 49 49 (1) else it is RGB 48 48 48 (0).

• Any tile can have any decimal value for any ASCII value.
• values 0 - 9, 11 - 42, 47, 59, and 64 appear as B (appears as a space in words, but only when selected twice / after a new letter selected) [can be blacked out]
- note that 48 is a B with 9 ice, 50 is a B with 1 ice, 51 is a B with 2 ice... 57 is a B with 8 ice. 
• values 91 - 93, 95 - 117, and 117 - 127 appear as B (appears as a space in words, but only when selected twice / after a new letter selected) [can be blacked out]
• values 128 - 255 appear as A (appears in words) [can be blacked out]
